The wrapped set created by the application of a selector can be treated as a JavaScript array for convenience. It is particularly useful to use array indexing to directly access elements within the wrapped set. For example: var element = bl("button",0); will set the variable element to the first element in the matched set.

We are presented with a the settings file called gitolite.rc in VIM editor which doesn't need to be edited. To exit VIM type :q! and press enter.

Now from your local machine clone the gitolite-admin repo from your server to your local machine. This will use the private key that we previously added to the known hosts file to access your remote server over SSH by matching the public key.

The cloned gitolite-admin repositorie is where we add SSH keys and set authorization for our users. Now let's edit the gitolite-admin/conf/gitolite.conf to include read rights for a new user called 'johndoe' for the testing.git repository that Gitolite generated for us.

Next we will generate a public and private SSH key for 'jonhdoe', something John should do himself of course. But we're being nice to John, because that's how we are. Let's say we are busy and let John do this himself. There are many options to do this on multiple platforms. John could download git and use the 'git gui' that comes with is, this way his private key will be used automatically to make the requests. Another option is 'PuTTYgen' that comes with 'PuTTY' the generated key can then be added to memory with 'Pageant' that also comes with 'PuTTY'. These are a few options to generate and add our SSH keys to make requests, but there are many more and git clients provide their own too. But for Mac and Linux we will use 'ssh-keygen' and add it to our known hosts.

Now that we have John's private and public key or maybe just his public key because he wanted to do it himself (what a guy). We will need to add his public key to the gitolite-admin/keydir directory. So now we have the following: an edited gitolite.conf file that gives our good old friend with the username 'johndoe' read rights on the testing.git repository and his public key johndoe.pub in the keydir directory. We are now all setup to push this to the repository on our remote server. This action is what will make John happy, because when this is done he is able to clone our repository and contribute on our beautifully written code.

Now we have successfully added John and gave him right to clone and push to the repo, but he was bummed out that it was the testing. So now let's setup a new repository to work on with John.
